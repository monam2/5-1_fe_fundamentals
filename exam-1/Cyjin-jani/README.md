# Exam 1: 다중 필터 상품 목록 구현

## 문제 이해

- 이커머스 서비스의 **상품 목록 페이지** 구현.
- 상품의 필터링, 정렬, 검색 기능 (auto complete)를 포함
- API 요청에 대한 응답 처리 시 로딩, 에러 처리 필요.

## 사용한 기술

- **TypeScript** : 필수 조건
- **React** : 필수 조건
- **TailwindCSS** : 가장 익숙한 스타일링 도구. 평가대상에서 제외라서 자유롭게 선택.
- **TanstackQuery** : 서버 상태 관리를 위해 도입. 커스텀 훅을 사용하여 컴포넌트의 역할과 책임 분리.
- **ReactErrorBoundary** : 선언적으로 로딩 상태, 에러 상태 등을 관리하기 위해 도입

## 필수 요구사항

### 상품 목록
- [X] API에서 상품 데이터를 불러와 카드 형태로 렌더링 (이미지, 상품명, 가격, 카테고리, 평점)
- [X] 상품명이 길 경우에도 레이아웃이 깨지지 않도록 처리

### 필터링
- [X] **카테고리 필터**: 체크박스 다중 선택 (shoes, tops, bottoms, accessories)
- [X] **키워드 검색**: 상품명 검색 (디바운싱 적용)

### 검색 자동완성
- [X] 검색 input에 글자를 입력하면 자동완성 API를 호출하여 **드롭다운으로 제안 목록 표시**
- [X] 제안 항목 클릭 시 해당 검색어로 검색 실행
- [X] 적절한 시점에 드롭다운 닫기 (선택, 외부 클릭, Esc 등)

### 정렬
- [X] 가격 낮은순 / 가격 높은순 / 최신순 / 평점순

### URL 동기화
- [X] 필터/정렬/검색 상태가 URL 쿼리스트링에 반영
- [X] 새로고침해도 필터 상태 유지
- [X] URL을 복사해서 공유하면 같은 필터 상태로 열림

### 필터 초기화
- [X] "전체 초기화" 버튼으로 모든 필터를 기본값으로 리셋

### 상태 처리
- [X] 로딩 중 표시
- [X] API 에러 시 사용자 안내 + 재시도 가능
- [X] 필터 결과가 없을 때 빈 상태 안내

## PR 작성 가이드

PR 본문에 아래 항목들을 포함해주세요. 코드만큼이나 **설계 판단의 근거**가 중요합니다.

### 1. 필터링 전략
- 서버 사이드 필터링
- 선택 이유
    - 데이터 수가 아주 많지 않은 경우에 적합
    - 필터 시마다 서버 데이터를 조회하므로 최신 데이터를 보여주는 데 유리함
      - 미리 받아서 보여주는 경우에, 일정 간격마다 data를 갱신해주지 않으면 최신 데이터를 바로 보기 어려울 수 있음 
    - 반면에, 데이터가 자주 변경되지 않는 데이터의 경우라면 필터링 마다 최신 정보를 불러오지 않아도 되므로 클라이언트 사이드가 더 유리할 수 있음.
      - 서버 요청 횟수가 줄고, 비용 감소

### 2. 상태 관리 전략
- URL 파라미터로 상태관리
- 불필요한 메모리를 사용하지 않아도 되며, 요구사항 구현에 용이하다고 판단함.
- 메모리에서 관리하면, 해당 상태를 유일한 소스(진실의 원천?)로 보장을 시켜야 하는데, 단순히 URL 파라미터를 쓰면 쉽게 해결됨.

### 3. 컴포넌트 구조
- product 리스트 관련이라서 크게 products와 shadcn ui 사용 때문에 ui 폴더 이렇게 두 가지로 분리.
- 현재로서는 과하게 분리할 필요가 없다고 느낌.
- 약간은 colocation을 생각했으나, components와 hooks를 분리시켜버렸기 때문에 조금은 애매한 감이 있다고 생각중.

### 4. 검색 및 자동완성 구현
- SearchInput 컴포넌트 내부에서 사용. (UI 레이어..?)
- 500ms로 처리. (너무 짧으면 불필요하게 요청이 많아지고 너무 길면 반응이 느리다고 판단..)
- useAutocompleteDropdown 훅을 이용해서 열림 상태와 외부 클릭 감지를 통해 닫을 수 있도록 관리함


### 5. 에러/재시도 처리
- 리스트 에러는 에러바운더리로 처리. autoComplete 에러의 경우 에러바운더리로 처리하면 오히려 안좋은 UX라 생각하여 지엽적인 에러인 autoComplete은 따로 에러를 처리하도록 함.
- 에러 바운더리의 reset을 이용.
  - 좀 더 정확히는 useQueryErrorResetBoundary를 사용함.

### 6. 과제를 수행하는데 어려움은 없었나요?
- 편리한 도구 (ai, tab 자동완성 등) 없이 순수하게 구현해보려고 했으나, 매우 어려웠음.
- 나중에 결국 시간 부족 이슈로... AI를 곁들여서 코딩을 하게 됨.
  - 내가 생각하는 나름의 괜찮은 설계 혹은 구현 방식을 먼저 AI에게 설명하고, 구현 후 더 나은 방향을 고민해보는..
    - 그러나 이마저도 너무 AI 없이 푸는 것에 집착하여 시간을 많이 써서 실패.
- biome.json이 있어서 biome을 써봤는데, 이것 때문에 환경 설정 충돌이나 이슈가 많아 해결하는 시간이 꽤 걸림.
