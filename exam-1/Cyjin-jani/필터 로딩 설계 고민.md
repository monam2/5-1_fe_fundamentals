# 필터 로딩 UI 아키텍처 고민

## 배경

필터(카테고리, 키워드, 정렬) 변경 시 로딩 상태를 사용자에게 시각적으로 알려주는 UI가 필요했음.
현재 프로젝트는 `useSuspenseQuery` + `useTransition` 조합으로 구성되어 있었음.

---

## 핵심 문제: `isPending` 소유권

`useProductFilters` 내부에서 `useTransition`을 사용해 `isPending`을 관리하고 있었음.

```ts
// useProductFilters.ts
const [isPending, startTransition] = useTransition();
```

`useTransition`은 **호출한 컴포넌트 인스턴스에 종속된 로컬 상태**다.
즉, 동일한 hook을 여러 컴포넌트에서 각각 호출하면 **독립된 `isPending` 인스턴스**가 생성되는 문제.

```
ProductFilters → useProductFilters() → 인스턴스 A → isPending_A (startTransition_A 호출 시 true)
ProductList    → useProductFilters() → 인스턴스 B → isPending_B (항상 false ← 버그)
```

카테고리를 클릭하면 `ProductFilters`의 `toggleCategory` → `startTransition_A`가 실행되므로,
`ProductList`에서 `useProductFilters().isPending`을 쓰면 **항상 `false`를 반환하는 죽은 상태**가 되어버림.

---

## 검토한 대안들

### 대안 1: `useSuspenseQuery` → `useQuery` + `keepPreviousData`

`useQuery`로 교체하면 `isFetching` 상태를 직접 소비할 수 있고, `isPending` 공유 문제가 사라짐.

```ts
const { data, isFetching } = useQuery({
  queryKey: [...],
  queryFn: fetchProducts,
  placeholderData: keepPreviousData,
});
```

**장점**
- `isPending` 소유권 문제가 구조적으로 사라짐.
- `isFetching`이 `useProducts` 안에 귀속되어 책임이 명확하다고 생각.
- `useTransition`을 제거할 수 있어 `useProductFilters`가 단순해짐.

**단점**
- `useSuspenseQuery`의 핵심 보장(`data`가 항상 정의됨)을 포기해야 함.
- `data`가 `TData | undefined`가 되어 코드 전반에 방어 코드가 늘어나게 됨.
- 초기 로딩 처리를 직접 해야 하는 불편함 (`if (!data) return ...`)

**결론**: `useSuspenseQuery`의 타입 안전성 이점이 크다고 생각하여 useSuspenseQuery를 유지해보기로 함.

---

### 대안 2: `Products`에서 단일 호출 + prop drilling

`useProductFilters`를 `Products`에서만 호출하고, 필요한 값을 자식에게 prop으로 전달.

```tsx
export const Products = () => {
  const { filters, isPending, hasActiveFilters, setKeyword, ... } = useProductFilters();

  return (
    <>
      <ProductFilters filters={filters} isPending={isPending} ... />
      <ProductList filters={filters} isPending={isPending} />
    </>
  );
};
```

**장점**
- `isPending`의 소유권이 단일 호출 지점에 귀속되어 상태 불일치가 구조적으로 불가능..
- Context 도입 없이 구현 가능하긴 함.

**단점**
- `Products`가 단순한 레이아웃 컴포넌트여야 하는데, `useProductFilters`의 모든 반환값을 알고 분배하는 **중간 관리자** 같은 역할을 함.
- `useProductFilters`의 인터페이스가 변경될 때 `Products`도 반드시 수정해야 함. (불필요한 의존성)
- prop의 수가 많아 컴포넌트 시그니처가 복잡해짐.

**결론**: 소유권 문제는 해결되지만 `Products`의 책임 과부하 문제가 생김..

---

### 최종 선택: Context API

`useProductFilters`를 단 한 번 호출하는 Provider를 만들고, 필요한 컴포넌트가 직접 접근해서 사용.

```
ProductFilterProvider (useProductFilters 단일 호출 → Context에 올림)
  └── Products
        ├── ProductFilters → useProductFilterContext()로 직접 소비
        └── ProductList   → useProductFilterContext()로 직접 소비
```

**장점**
- `isPending`의 소유권이 Provider 단 하나에 귀속됨. (상태 불일치 원천 차단)
- `Products`는 Provider로 감싸기만 하면 되어 책임이 명확하다 생각.
- `useProductFilters` 인터페이스가 변경되어도 `Products`는 수정 불필요.
- `useProductFilterContext`에 Provider 외부 사용 시 에러를 던지는 가드가 있어 잘못된 사용을 방지 가능.

**단점**
- Context를 구독하는 모든 컴포넌트가 값 변경 시 리렌더링됨.
- 현재 규모에서는 약간 과한 설계일 수 있음.

**결론**: 구조적 명확성과 확장성을 우선시하여 채택.

---

## `useTransition` + `useSuspenseQuery` 조합에 대한 고민..

- `startTransition`으로 감싼 상태 변경은 React가 **현재 UI를 유지하면서 백그라운드에서 새 렌더링을 준비**하는데,
- 덕분에 필터 변경 시 Suspense fallback(skeleton)이 발동되지 않고, 이전 리스트를 보여주면서 백그라운드 fetch가 진행됨..
- 다만 위에서 서술한 `isPending` 같은 문제가 발생할 수 있다고 생각.
- 실무에서라면 오히려 `useQuery` + `isFetching`로 단순하게 처리했을 것 같기도 함...
